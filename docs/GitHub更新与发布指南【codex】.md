# GitHub 更新与发布指南

这份文档供项目维护者在以后说“同步到 GitHub”或准备新版本时使用。目标只有两个：公开版本与日常开发版本始终相互独立；每次发布都能用同一套步骤快速检查、推送和回滚。

## 1. 三个仓库必须保持分离

| 仓库目录名 | 用途 | 不能做的事 |
| --- | --- | --- |
| `one-holy-bible` | 日常开发、卡片工作台、源数据处理和实验功能 | 不直接作为 GitHub 发布目录，不从脏工作区整包复制或推送 |
| `one-holy-bible-github` | GitHub 公开版本的唯一发布目录 | 不承接日常开发，不放私有源文件、临时文件或本机路径 |
| `one-holy-bible-assets` | 公开图片资源仓库 | 不放应用源码，不覆盖已经发布并被引用的同名资源 |

以后收到“同步到 GitHub”的指令时，默认只在 `one-holy-bible-github` 中整理、验证、提交和推送。开发仓库只能作为来源，不能把开发仓库的分支或远端临时改成公开仓库。

## 2. 更新前先确定发布边界

每次开始前先回答以下问题：

- 本次从开发版同步哪些已经完成的功能、经文、解释文字卡或图片卡？
- 本次明确不发布哪些实验功能、工作台数据、私有资料或未完成内容？
- 来源是否已经形成可复现的提交或稳定快照？不要从仍在变化的未提交文件直接制作公开版。
- 这是补丁、兼容更新还是新增功能？据此选择下一版本号。

建议使用语义化版本：

- 修复公开版本而不改变功能范围：`v0.2.1`；
- 增加明显的新功能或新公开内容：`v0.3.0`；
- 存在不兼容的大改动：再考虑提升主版本号。

已经发布的 tag 应视为不可变快照。当前 `v0.2.0` tag 保留在原始发布提交；后续维护应使用新提交和新版本号，不移动旧 tag，也不强制覆盖远端历史。

## 3. 推荐的同步顺序

1. 在开发仓库确认来源提交、功能范围和内容范围。
2. 在独立的 GitHub 仓库中建立发布分支，只移植本次确定公开的改动。
3. 如果涉及新图片，先处理并推送资源仓库。
4. 重新生成公开数据、图片清单、搜索索引和发布清单。
5. 更新版本号、说明文档、数据来源和第三方声明。
6. 完成本地验证，再提交和推送 GitHub 公开仓库。
7. 检查 GitHub Actions、公开图片 URL 和最终仓库状态。

不要使用“把开发目录整体覆盖到 GitHub 目录”的方式同步。应当按文件和功能有选择地移植，这样开发中的临时状态不会进入公开版本。

## 4. 图片资源的发布规则

公开应用仓库不直接保存全部图片二进制文件。图片应位于 `one-holy-bible-assets`，应用数据通过资源描述符引用它们。

发布新图片时必须遵守：

- 资源文件名使用内容哈希，避免同名覆盖和缓存混乱；
- 先把图片推送到资源仓库，再生成公开应用中的 URL；
- URL 应固定到资源仓库的具体 commit，而不是会继续变化的分支地址；
- 每个图片描述符应保留 SHA-256、字节数、MIME 类型、宽度和高度；
- 同一张图片被多张卡片引用时，只存一份二进制资源；
- 更新后核对图片卡数量、唯一图片数量、资源清单数量和哈希是否一致；
- 至少抽查一个新增图片 URL 能公开访问，且返回的文件哈希与清单一致。

如果图片卡突然消失，优先检查 `imageCards` 是否仍在各书卷数据中、`asset-manifest.json` 是否包含对应哈希，以及 URL 是否指向已经推送的资源 commit。不要因为代码仓库里没有 PNG 文件就误判为图片功能缺失。

## 5. 经文与文字卡的数据规则

公开数据继续按 66 卷拆分并按需加载。更新时应保持现有公开 schema，不把开发工作台的内部结构直接暴露出去。

必须核对：

- 66 卷书卷文件和清单一一对应；
- CUV、KJV 的书卷、章节和经文数量与发布清单一致；
- `textCards` 与 `imageCards` 的引用范围有效；
- 搜索索引与书卷数据来自同一次生成结果；
- 清单中的文件大小、SHA-256 和统计数字与实际文件一致；
- JSON 是严格格式，不含注释、尾随逗号或 `NaN`；
- 不含本机绝对路径、私有 URL、密钥、会话信息或仅供开发使用的元数据。

## 6. 每次发布需要同步更新的内容

根据本次范围逐项检查：

- `package.json` 和 `package-lock.json` 中的版本号；
- `README.md` 的版本范围、功能、数据统计和验证命令；
- `DATA_SOURCES.md` 与 `THIRD_PARTY_NOTICES.md`；
- `public/data/manifest.json`、`public/data/asset-manifest.json` 和相关发布清单；
- 66 卷数据包与搜索索引；
- `docs/releases/` 中的新版本发布说明；
- `docs/verification/` 中的新版本验证记录；
- GitHub Actions 所调用的脚本名称和 `package.json` 中实际存在的脚本。

文档中凡是写了具体版本号、数量、资源 commit 或校验结果，都要与本次产物重新核对，不能只复制上一版文字。

## 7. 推送前的固定验证

在 `one-holy-bible-github` 中执行：

```bash
npm run validate:public-data
npm run validate:public-repository
npm run validate:public-release
npm test
npx tsc --noEmit
npx tsc -p tsconfig.node.json --noEmit
npm run build
npm audit --omit=dev
```

然后检查实际差异和仓库身份：

```bash
git status -sb
git diff --check
git diff --stat
git remote -v
git log -5 --oneline --decorate
```

只暂存本次确认过的文件，避免无条件使用 `git add -A`。提交前再次查看暂存差异：

```bash
git diff --cached --stat
git diff --cached
```

如果本次改变了可见界面或交互，还需要在真正的公开构建运行面走一遍主路径，至少确认书卷切换、经文显示、文字卡和图片卡能够正常加载。

## 8. 推送后的检查

推送 `main` 后不要立即把任务视为完成，还要确认：

- 本地 `main`、`origin/main` 和预期提交一致；
- GitHub Actions 最新运行成功；
- GitHub 仓库首页的版本说明和链接正常；
- 至少一个图片资源 URL 可公开访问；
- 新版本 tag（如本次创建）指向经过验证的发布提交；
- 开发仓库和资源仓库没有被意外提交应用公开版的无关文件。

## 9. 出现问题时如何处理

- **CI 失败：** 先读失败步骤和日志，在新提交中修复；不要删除失败记录掩盖问题。
- **图片不显示：** 按“书卷数据 → 图片哈希 → 资源清单 → 固定 commit URL → 远端文件”的顺序排查。
- **数据校验失败：** 重新生成整套相关产物，不要手工只改统计数字或哈希。
- **推错仓库：** 立即停止后续推送，确认三个仓库的目录、分支和 remote，再制定修复方案。
- **发布后发现错误：** 优先追加修复提交并发布补丁版本；不要对公开历史执行 destructive reset 或 force push。
- **旧 tag 有问题：** 保留原 tag，使用新版本说明修正。只有在明确评估外部影响并得到确认后，才考虑移动 tag。

## 10. 最短发布检查表

```text
[ ] 当前操作目录是 one-holy-bible-github
[ ] 开发来源已提交或形成稳定快照
[ ] 本次公开与排除范围已写清楚
[ ] 新图片已先推送到 one-holy-bible-assets
[ ] 图片 URL 固定到资源 commit
[ ] 66 卷、经文、文字卡、图片卡和搜索索引统计一致
[ ] 版本号、README、来源声明、发布说明和验证记录已更新
[ ] 全部固定验证命令通过
[ ] 已检查 git status、diff、remote 和提交目标
[ ] origin/main 已推送且 GitHub Actions 成功
[ ] 公开页面和图片主路径已抽查
[ ] 旧 tag 未被移动，开发仓库未被污染
```

只要这张检查表没有全部满足，就应把状态表述为“已准备但尚未完成发布”，而不是“已经同步完成”。
